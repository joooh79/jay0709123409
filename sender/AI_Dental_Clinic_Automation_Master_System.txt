AI DENTAL CLINIC AUTOMATION — MASTER SYSTEM
FINAL

This document is the bible of the AI Dental Clinic Automation subsystem.

It is not the constitution of the whole project.
It operates under:
- Foundation Canon
- Document Hierarchy Definition
- SAG governance

Its purpose is to:
- fix automation structure
- define automation workflow and constraints
- define canonical JSON behavior
- define sender / webhook / Make / Airtable architecture
- define the higher-level lock for sender-stage and live-state gating behavior
- define where active sender runtime identity and implementation reference are recorded when relevant
- embed Airtable structure and option mapping rules into one locked automation source

==================================================
1. SYSTEM GOAL
==================================================

Receive doctor voice/text briefing
→ generate canonical JSON
→ sender transforms canonical JSON into Airtable-ready fragments
→ webhook delivers to one integrated Make scenario
→ Airtable stores the structured records

==================================================
2. CORE ARCHITECTURE
==================================================

- input: doctor briefing (voice / text)
- processing: AI → canonical JSON generation
- transformation: sender → fragment transformation
- delivery: webhook → one Make scenario
- storage: Airtable
- live-state verification surface when relevant: current-state MCP

Fixed components:
- sender baseline: 1 primary baseline governed by operational protocol
- webhook: 1
- make scenario: 1

==================================================
3. WORKFLOW MODES
==================================================

The automation system allows only 3 modes:
1. new_patient_new_visit
2. existing_patient_new_visit
3. existing_visit_update

==================================================
4. MODE DECISION LOGIC
==================================================

- new patient + new visit → new_patient_new_visit
- existing patient + new visit → existing_patient_new_visit
- existing patient + existing visit update → existing_visit_update

If one axis is unclear:
- JSON generation prohibited
- clarification required first


==================================================
4A. ROUTE EXECUTION VALIDATION RULE
==================================================

Workflow mode determines the intended route candidate,
but it does not by itself authorize blind downstream execution.

Required rule:
- sender must validate required route-identifying fields before delivery
- Make must perform execution safety checks before route entry or write
- missing route-identifying requirements may block, re-check, or correction-route the case
- mode resolution alone is not sufficient justification for blind downstream execution

==================================================
4B. EXECUTION-READY SAMPLE GATING RULE
==================================================

If an execution-ready sample JSON, send-ready sample JSON, or validation sample depends on live Airtable state for correct meaning, safe routing, or meaningful update validation, current-state gating must be checked before the sample is treated as execution-ready.

Lock:
- sample readiness must not be declared from workflow.mode interpretation alone when live-state gating is relevant
- explanatory route reasoning does not replace current-state validation
- live-state-dependent sample approval requires the relevant Airtable precheck first

Default efficient current-state rule:
- patient-only gate → patient:{patient_id}
- patient + visit_date gate → bundle:{patient_id}:{visit_date}
- existing_visit_update gate → confirm target visit existence first

==================================================
5. VISIT / PATIENT SAFETY RULES
==================================================

- patient_id alone must not determine patient status
- Dentweb patient_id may exist for both new and existing patients
- if patient status is unclear, ask first
- if visit status is unclear, ask first
- for existing_visit_update, prefer visit date before visit_id
- visit_id in normal workflow is derived, not directly demanded from the doctor



==================================================
5A. VISIT EVENT VS CURRENT BRIEFING INTENT CLARIFICATION RULE
==================================================

The actual patient visit event and the current doctor briefing moment are separate.

Workflow mode classification must consider:
1. patient status
2. visit recording intent

Interpretation lock:
- new visit means the doctor is now recording that visit event for the first time in the system
- existing_visit_update means the doctor is now updating, correcting, or completing a visit event that has already been recorded
- existing_visit_update does not mean the patient physically visited again on that same date

Operational consequence:
- same-date Airtable existence is important routing evidence
- but same-date existence alone does not fully define the doctor’s current briefing intent
- current-state existence and current briefing intent must not be collapsed into one concept

7. FINDINGS STRUCTURE
==================================================

Findings are represented through:
- findings_present
- findings_records[]

Conceptual clinical branches remain:
- pre_op
- radiographic
- operative
- diagnosis
- treatment_plan
- doctor_reasoning

Interpretation:
- findings_present preserves branch-level presence flags
- findings_records[] preserves tooth-level clinical rows
- final canonical JSON must not use legacy top-level branch sections such as pre_op / radiographic / operative / diagnosis / treatment_plan / doctor_reasoning as final output containers


==================================================
8. FINDINGS EXECUTION RULE
==================================================

Downstream findings execution is row-based under findings_records[] and branch-routed after sender transformation.

Required rule:
- findings_records[] must contain at least one valid clinical row before execution is allowed
- a branch may execute only when sender-transformed output for that branch contains at least one valid Airtable-ready field derived from findings_records[]
- reserved sender-managed fields alone do not justify execution

Reserved sender-managed fields:
- Record name
- Visit ID
- Tooth number

Otherwise:
- branch execution prohibited


==================================================
9. MINIMUM FINDING RULE
==================================================

If findings_records[] is empty or effectively reserved-only:
- JSON generation prohibited
- briefing considered insufficient

Minimum acceptable condition:
- findings_records[] contains at least 1 valid clinical row with at least 1 real clinical field beyond reserved sender-managed fields

==================================================
10. PATIENT ID RULE
==================================================

patients.patient_id:
- exact 6-digit Dentweb chart number
- no prefix
- no suffix
- no format alteration
- keep leading zero when present

==================================================
11. VISIT ID RULE
==================================================

visit_id format:
VISIT-{patient_id}-{YYYYMMDD}

Assumption:
- one visit per patient per date

For existing_visit_update:
- workflow.target_visit_date should be filled when known
- workflow.target_visit_id may remain empty
- visits.visit_id should match the target visit when known

==================================================
12. VISIT DUPLICATION RULE
==================================================

Before creating a new visit:
- search by visit_id

If the same visit_id already exists:
- Create prohibited
- route to existing_visit_update or error handling

==================================================
12A. SAME-DATE VISIT CORRECTION RULE
==================================================

If:
- visit_intent_claim = existing_visit_update
- and same_date_visit_exists = true

Then:
- gate_decision = NORMAL_PASS
- same-date correction_required must not be triggered

If:
- visit_intent_claim = new_visit
- and same_date_visit_exists = true
- and doctor_confirmed_correction is empty

Then:
- gate_decision = CORRECTION_REQUIRED

If:
- visit_intent_claim = new_visit
- and same_date_visit_exists = true
- and doctor_confirmed_correction = true

Then:
- gate_decision = NORMAL_PASS

Interpretation lock:
- same-date correction is for intent/search-result mismatch handling
- it must not override an already-matching existing_visit_update path

==================================================
12B. EXISTING_VISIT_UPDATE FINDINGS WRITE RULE
==================================================

During existing_visit_update:
- findings write must update/upsert the matching existing branch row
- matching existing branch row must not be treated as a new append target merely because the same Visit ID is being processed again
- duplicate child-row append behavior is prohibited in validated existing visit update flow

Deterministic identity rule:
- update targeting should follow the already-resolved existing visit context
- incoming duplicate visit_id must not be treated as justification for creating duplicate findings rows

==================================================
12C. SUCCESS RESPONSE HANDLING RULE
==================================================

When write succeeds:
- success response should be generated through central gate/result response handling
- explicit success JSON may be returned to sender
- branch-local success-response multiplication should be avoided

Preferred direction:
- central decision/result shaping
- central webhook response return
- no unnecessary per-branch response duplication

==================================================
12D. FINDINGS-CENTERED EXISTING_VISIT_UPDATE VALIDATION RULE
==================================================

For existing_visit_update validation, the clinically meaningful write target is the findings-level current state, not only route passage or high-level success messaging.

Validation consequence:
- when findings-level change is the purpose of the test, current stored findings state should be checked first
- validation should focus on whether the intended branch/tooth/record-level change landed without duplicate append or wrong-target overwrite

==================================================
12E. ACCEPTED_NO_JSON SUCCESS SHAPE RULE
==================================================

Under the current Make write-path architecture, the normal write path may return an accepted transport response without structured JSON.

Interpretation:
- accepted_no_json may represent a normal write-path success shape
- it must not be treated as failure by default when Make execution success and Airtable write success are consistent with the current validated architecture

==================================================
13. PATIENT SEARCH FAIL RULE
==================================================

If patient search returns 0 in:
- existing_patient_new_visit
- existing_visit_update

Then:
- automatic fallback to new_patient_new_visit prohibited
- re-check or error handling required

==================================================
13A. ONE-TIME PATIENT ID RECHECK RULE
==================================================

If:
- patient_status_claim = existing_patient
- and patient search result = 0

Then:
- automatic fallback remains prohibited
- automatic new-patient conversion remains prohibited
- write remains prohibited
- preserved payload may be maintained
- one-time patient_id recheck may be returned

First fail behavior:
- correction_required response allowed
- reason_code = PATIENT_NOT_FOUND_RECHECK_REQUIRED
- write_allowed = false
- resend_allowed = true

Second fail behavior:
- hard_stop required
- reason_code = PATIENT_NOT_FOUND_RECHECK_FAILED
- write_allowed = false
- resend_allowed = false

Lock:
- this is not automatic fallback
- this is not automatic patient creation
- this is one narrow identity recheck layer only

==================================================
13B. PATIENT RECHECK RESEND PRESERVATION RULE
==================================================

When sender performs patient_id re-entry resend for the allowed one-time recheck path:
- clinical content must be preserved
- correction-only re-entry should be supported without re-briefing the full case
- workflow.patient_recheck_attempted must reflect the recheck state
- patients.patient_id, visits.visit_id, findings_records[].visit_id, and findings_records[].record_name must remain consistency-safe after re-entry resend
- after second-fail hard_stop, resend UI must not remain available for continued retry

==================================================
13C. BASELINE-FIRST UPDATE VALIDATION RULE
==================================================

When an update-style execution test or execution-ready update sample is being prepared:

1. current target state should be checked first
2. the before-state should be preserved as the validation baseline
3. no-op payloads must not be mistaken for meaningful update validation
4. after execution, the same target should be re-checked against the preserved baseline

Interpretation lock:
- successful route passage alone is not sufficient evidence of meaningful update validation
- update validation should confirm that the intended stored change actually landed

==================================================
14. JSON GENERATION GATE RULE
==================================================

JSON may be generated only when all required conditions are satisfied:

1. patient_id exists
2. patient status resolved
3. visit status resolved
4. relevant date exists
5. findings_records[] contains at least one valid clinical row

Otherwise:
- JSON generation prohibited
- clarification required first

------------------------------
19.2A Findings Record Identity Rule
------------------------------

Findings are tooth-level child records under one visit.
A single visit may contain multiple treated teeth.
Therefore, each findings table may contain multiple records under the same Visit ID,
but each record must represent one tooth in one findings branch.

Record name rule
- Record name = {Visit ID}-{Tooth number}-{BRANCH CODE}

Branch codes
- Pre-op Clinical Findings = PRE
- Radiographic Findings = RAD
- Operative Findings = OP
- Diagnosis = DX
- Treatment Plan = PLAN
- Doctor Reasoning = DR

Examples
- VISIT-901710-20260321-36-PRE
- VISIT-901710-20260321-36-RAD
- VISIT-901710-20260321-36-OP
- VISIT-901710-20260321-36-DX
- VISIT-901710-20260321-36-PLAN
- VISIT-901710-20260321-36-DR

Uniqueness rule
- Within the same findings table, the combination of Visit ID + Tooth number + Branch Code must be unique.
- Same visit + same tooth + same findings branch must not create duplicate records.
- Multi-tooth cases are handled by creating separate tooth-level child records under the same Visit ID.

GENERAL HARD STOPS
- If no exact option exists, use "".
- If certainty does not match an available option, use "".
- If the phrase is clinically meaningful but not canon-aligned, use "" and preserve nuance in reasoning_notes when appropriate.
- Do not output approximate variants in option fields.
